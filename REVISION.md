# MEDISENS Panel Revision Sprint

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this sprint task-by-task. Patient account work is explicitly deferred.

**Goal:** Address panel feedback for medical staff workflows without changing role strings, replacing the Vite multi-page app, or making destructive database changes.

**Architecture:** Keep the current `src/` structure. Add itemization and transaction-history helpers under `src/features/patients/`, then surface the data through shared patient history UI. Store new vaccine detail in existing FHSIS `data_fields` JSON to avoid schema changes during defense hardening.

**Tech Stack:** Vite, React, TypeScript, Tailwind utility classes, Supabase.

---

## Scope Decisions

- [x] Patient accounts are ON HOLD.
- [x] Medical staff workflows remain the priority.
- [x] No destructive database changes.
- [x] No role string renames.
- [x] Preserve existing Supabase tables and current workflow paths.

## Deferred Feature

### Patient Accounts

Patient-facing accounts should be implemented after the medical staff system is finalized. This requires a separate design pass for patient auth, RLS policies, account linking, password recovery, patient-visible history scope, and consent/privacy rules.

## Current Storage Audit

- Ailments / complaints:
  - `initial_consultation.chief_complaint`
  - `consultation.chief_complaints`
- Findings:
  - `lab_result.findings`
  - `consultation.assessment`
- Treatments:
  - `consultation.medication_treatment`
  - `consultation.management_treatment`
  - `follow_up.medication_treatment`
- Tests:
  - `lab_request` boolean columns such as `is_cbc`, `is_urinalysis`, `is_xray`, plus `others`
- Prescriptions:
  - `prescription.rx_content` JSON string containing medication items
- Vaccines:
  - Existing child vaccine fields are currently stored in `fhsis_logs.data_fields`, including `bcg_date`
  - Sprint adds `vaccine_records` array inside `data_fields` for multiple vaccine entries
- Transaction history:
  - Currently scattered across patient registration, consent, initial consultation, consultation, lab request/result, prescription, FHSIS, and follow-up tables

## Future Migration Plan

No migration is required for this sprint. For a future normalized implementation, add these tables instead of overloading text columns:

```sql
-- proposed only, not applied in this sprint
create table patient_ailments (
  id bigint generated by default as identity primary key,
  patient_id bigint not null references patients(id),
  source_table text not null,
  source_id bigint,
  label text not null,
  notes text,
  recorded_at timestamptz default now()
);

create table patient_treatments (
  id bigint generated by default as identity primary key,
  patient_id bigint not null references patients(id),
  consultation_id bigint,
  treatment_type text not null,
  description text not null,
  recorded_at timestamptz default now()
);

create table patient_vaccines (
  id bigint generated by default as identity primary key,
  patient_id bigint not null references patients(id),
  vaccine_name text not null,
  dose_label text,
  date_given date,
  remarks text,
  source_log_id bigint,
  recorded_at timestamptz default now()
);
```

If this migration is approved later, add RLS policies and backfill from existing `consultation`, `lab_request`, `prescription`, and `fhsis_logs` records.
